# 婚礼请柬竞品调研报告

> 调研日期：2026-05-12
> 调研范围：国内主流模板平台 / 国外婚礼网站平台 / 开源参考项目 / Awwwards 灵感站
> 目标：为后续设计自研婚礼请柬 H5 提供事实依据

## 一、共性结构（常见页面顺序）

按出场顺序排列，这是国内婚贝/婚礼纪/婚礼乎/MICO 类 H5 请柬，以及国外 Joy/Greenvelope/Riley & Grey 共同收敛的结构：

| # | 页面 | 1 行说明 | 是否常见 |
|---|---|---|---|
| 1 | 启动/封面页 | 初始 loading + 标题 + 大字日期 + 引导手势（点击进入 / 上滑） | 必备 |
| 2 | 信封开启 / 入场动画 | 模拟拆信封、撕开蜡封、纸张翻开 | 国内中端以上必备 |
| 3 | 主视觉 Hero 页 | 大尺寸婚纱照 + 新人姓名 + Save the Date | 必备 |
| 4 | 我们的故事 / Love Story | 时间轴或长滚动 + 配图 + 短文案 | 高频 |
| 5 | 新人介绍 | 新郎新娘并列照片 + 简介 / MBTI / 兴趣 | 高频（年轻新人偏好） |
| 6 | 照片墙 / Gallery | 拼接排版 / 长滚动 / 轮播 / 灯箱 | 必备 |
| 7 | 婚礼信息 | 日期 + 倒计时 + 地点 + 流程时间表 | 必备 |
| 8 | 地图与导航 | 嵌入地图 + 一键唤起导航 + 交通方式 | 必备 |
| 9 | RSVP / 出席回执 | 出席与否 + 人数 + 饮食偏好 + 留言 | 国外必备，国内可选 |
| 10 | 留言祝福 / 弹幕 | 给新人留言 + 现场弹幕上墙 | 国内特色 |
| 11 | 礼物登记 / Registry | 礼物清单或现金账户链接 | 国外必备 |
| 12 | 出行 & 住宿 | 酒店推荐 + 班车 + Dress Code | 国外目的地婚礼必备 |
| 13 | 结尾感谢页 / Thank You | 致谢文案 + 转发分享按钮 | 必备 |

**节奏规律**：知乎《拒绝俗套》一文给出公认建议——**H5 6~8 页为佳**，再多就疲劳。国内主流单页全屏翻页，国外主流单页长滚动 + 顶部 Tab 导航。

## 二、单产品分析

### 1. 婚礼纪
- **类型**：H5 + APP + 小程序
- **页面顺序**：封面 → 入场动画 → 新人合照 → 故事 → 照片轮播 → 婚礼信息 → 地图 → 留言 → 致谢
- **动效特点**：标准翻页 + 入场缓动；模板内置动画时长固定
- **设计风格**：模板量大（号称上千套），分类细：囍嫁 / ins / 清新 / 网红 / 电影 / 轻奢 / 手绘 / 复古 / 卡通；中式以红+金为主
- **亮点**：模板分类是行业最细的；与婚礼酒店、相册一站式打通
- **缺点**：模板撞款率极高，发出去对方一眼能识别"婚礼纪模板"；个性化定制有限
- **参考价值**：⭐⭐⭐（最该参考的是模板分类与字段抽象）

### 2. 婚贝请柬
- **类型**：APP + H5（h5.hunbei.com）
- **页面顺序**：与婚礼纪类似，但额外提供「翻页 / 长页 / 海报 / MV」四种载体
- **动效特点**：翻页式比较多见；MV 模式有视频背景 + 字幕动画
- **设计风格**：以简洁大方为主，被评为"浪漫的微电影感"
- **亮点**：1) 模板风格一致性好（一套从头到尾不跳）；2) 同时支持 H5 + 视频 MV；3) 提供延期通知模板、父母邀请模板等细分场景
- **缺点**：付费模板才有高级动画
- **参考价值**：⭐⭐⭐⭐（"模板内一致性"是值得偷的设计原则）

### 3. 来吧请柬 / MICO 请柬
- **类型**：APP + 小程序为主
- **页面顺序**：封面 → 翻书动画 → 故事 → 照片墙 → 信息 → 地图 → 留言
- **动效特点**：以书本翻页 / 卷轴展开为差异化
- **设计风格**：偏 ins 简约 + 港风胶片
- **亮点**：MICO 主打"3D 翻书"和"一镜到底滚动"
- **缺点**：评测信息较少，用户量较婚礼纪/婚贝小一档
- **参考价值**：⭐⭐⭐（翻书 3D 已经"快烂大街"，不建议再做）

### 4. 易企秀（请柬模板）+ MAKA + 稿定设计
- **类型**：通用 H5 设计平台 → 婚礼是其中一个模板分类
- **页面顺序**：通用模板，无固定结构
- **动效特点**：覆盖元素级动画（飞入、淡入、放大）+ 翻页转场（滑动、覆盖、3D）；兼容微信 X5 内核
- **设计风格**：模板很杂；婚礼分类多，但偏"营销味"
- **亮点**：编辑器极强，所有元素可独立动画 / 配置触发；易企秀官方主推 H5 翻页
- **缺点**：模板风格不专业，"婚礼味"不够；含水印需付费
- **参考价值**：⭐⭐⭐⭐（动效能力的工程实现可参考；但 UI/UX 不要学）

### 5. 婚礼乎
- **类型**：H5 + APP
- **页面顺序**：标准翻页结构
- **动效特点**：以照片视频混合的 MV 模式见长（"不用剪辑的 MV"）
- **设计风格**：以"微电影感 + 字幕"为主
- **亮点**：MV 自动生成，新人只要填照片配文字
- **缺点**：模板偏老气，年轻人接受度一般
- **参考价值**：⭐⭐（MV 视频思路可参考，UI 不参考）

### 6. Greenvelope（国外）
- **类型**：网页（移动端响应式）
- **页面顺序**：信封封面动画 → 卡片正面 → 卡片背面 → RSVP → 礼物登记 → 出行
- **动效特点**：**核心招牌是"信封开启"动画**——模拟从邮箱拿出信封、打开蜡封、抽出卡片，过程精致；可配背景音乐
- **设计风格**：精致插画 / 烫金 / 水彩 / 极简，全部由独立插画师定制；强调"环保替代纸质"
- **亮点**：1) 信封动画是行业标杆；2) RSVP 追踪 = 类 CRM 体验，可看谁打开过、回了什么；3) 可发邮件 + SMS
- **缺点**：纯英文 + 海外服务，国内访问不顺；按邀请人数收费
- **参考价值**：⭐⭐⭐⭐⭐（**信封开场 + RSVP 流程**最值得抄）

### 7. Paperless Post（国外）
- **类型**：网页 + APP
- **页面顺序**：信封 → 卡片（正反面可翻） → 详情子页 → RSVP
- **动效特点**：信封打开动画 + 卡片**翻面动画**（正反面切换）；支持嵌入视频/GIF
- **设计风格**：现代简约、艺术插画、烫金
- **亮点**：1) **Flyer 模式**支持上传任意图片做底，叠加动画贴纸和文字效果；2) 支持邀请函中嵌入视频
- **缺点**：批量管理略弱
- **参考价值**：⭐⭐⭐⭐⭐（**"视频可嵌入" + "翻面动画"**值得借鉴）

### 8. Joy（withjoy.com）
- **类型**：网页 + APP（App + Web 双端体验一致）
- **页面顺序**：Cover → Story → Schedule → Travel → Registry → RSVP → FAQ → Wedding Party
- **动效特点**：**长滚动 + Tab 导航**两种共存；动画极克制，主打"高质感静态 + 漂亮字体"
- **设计风格**：600+ 模板，按色卡组织（Mocha / Olive / Papaya 等）；强调字体与配色
- **亮点**：1) **Smart RSVP**——按子事件分别 RSVP（彩排、晚宴、早午餐分开）；2) **Guest Q&A** 自动回答常见问题；3) 完全可改 CSS（让"会代码的人"自定义）；4) 与可打印请柬完全配套
- **缺点**：模板编辑器评价两极（部分用户觉得菜单嵌套乱）
- **参考价值**：⭐⭐⭐⭐⭐（**信息架构**是行业标杆——值得逐字段抄）

### 9. Riley & Grey（国外，高端）
- **类型**：纯网页，单页长滚动
- **页面顺序**：Hero（大幅婚纱照）→ Story → Details → Schedule → RSVP → Registry → 致谢
- **动效特点**：**Scrolling Curtains**（卷帘式滚动）——每屏像幕布一样向上"卷开"，下一屏从底部出现，类似剧院；可关闭改为 Tab 导航
- **设计风格**：杂志感、时尚大片质感；色卡偏柔和大地色 + 偶尔大胆撞色；字体精致衬线
- **亮点**：1) **Scrolling Curtains** 是行业难复制的招牌动效；2) 模板像 Vogue 编辑过的；3) 倾向用一张大照片 + 大字号衬线字
- **缺点**：付费且贵（$30+ 起）；模板量比 Joy 少
- **参考价值**：⭐⭐⭐⭐⭐（**视觉品味**最该学；卷帘动效可借鉴）

### 10. Zola / Minted / The Knot（国外，主流大众）
- **类型**：网页（一站式婚礼平台的子模块）
- **页面顺序**：标准结构
- **动效特点**：动效几乎为零，主打**好用 / 干净 / 信息明确**
- **设计风格**：模板多但风格保守；Minted 字体&配色搭配是亮点
- **亮点**：与礼物登记 / 客人管理 / 桌位安排深度集成
- **缺点**：动效弱、个性弱
- **参考价值**：⭐⭐⭐（信息架构可参考；视觉/动效不参考）

### 11. 开源参考项目

| 项目 | 技术栈 | 关键特征 | 参考价值 |
|---|---|---|---|
| **CahierX/wedding** | 微信小程序（uni-app/原生 JS） | 主打「**千人千面**」——每个好友收到独一无二页面（带昵称 + 关系定制文案）；含晚霞预测彩蛋；分享相册 | ⭐⭐⭐⭐ "一人一份"的思路是真正能让人记住的差异点 |

## 三、差异化亮点汇总（避免烂大街的元素）

| 产品 / 案例 | 创新点 | 用户反馈 | 可参考性 |
|---|---|---|---|
| Greenvelope | 信封拆封 + 蜡封打开动画 | 惊艳，仪式感强 | 高（**强烈推荐**） |
| Paperless Post | 卡片翻面 + 嵌入视频/GIF | 评价好 | 高 |
| Riley & Grey | Scrolling Curtains（卷帘滚动） | 高端感强 | 中（实现难度大但效果独特） |
| Joy | Smart RSVP（子事件分别回执）+ Guest Q&A | 解决真实痛点 | 高（信息架构必抄） |
| CahierX/wedding | 千人千面（一人一页） + 晚霞预测彩蛋 | 朋友打开惊喜 | **极高**（强烈推荐） |
| iammapping/wedding | 现场大屏 + 弹幕上墙 + 扫码签到 | 婚礼当天最炸 | 高（**做差异化的关键**） |
| MICO / 来吧 | 3D 翻书 | 第一次看惊艳，但已被滥用 | 低（**别做**） |
| 婚贝请柬 | 父母邀请 / 延期通知 / 出阁宴等子模板 | 解决细分需求 | 中 |
| 婚礼乎 | 自动 MV（不用剪辑） | 老人接受度高 | 中（思路可参考） |
| Weddingly-Free | URL 后缀 `/to:Name` 自动带名字 | 简单优雅 | 高（**几乎零成本实现千人千面**） |

**结论**：**翻书、纸张飘落、烟花、烫金 LOGO、心形浮动**这五样已经被国内模板用烂，不要再做。真正的差异化在三个方向——
1. **个性化文案 / 一人一页**（CahierX、Weddingly-Free）
2. **现场 + H5 联动**（iammapping）
3. **电影质感视频背景 + 极简字体**（Riley & Grey、Paperless Post）

## 五、手机端兼容性常见坑

来自微信开放社区、CSDN、掘金等中文技术社区高频问题汇总：

| # | 坑位 | 描述 | 推荐解法 |
|---|---|---|---|
| 1 | **微信浏览器音频 autoplay 禁用** | iOS 微信内 `<audio autoplay>` 直接失效；Android 部分机型也禁 | 用微信 jweixin SDK 的 `WeixinJSBridgeReady` 事件触发播放；或首屏放一个引导按钮"点击进入"用用户手势触发 |
| 2 | **iOS Safari `100vh` 实际超过可视区** | iOS Safari 把地址栏和底部工具栏算进 100vh，结果内容被裁；滚动时高度还会变 | 用 `100dvh`（动态视口）替代；旧版 fallback 用 JS 监听 `window.innerHeight` 写入 CSS 变量 `--vh` |
| 3 | **微信 X5 内核 video 行为差异** | Android 微信内 video 默认全屏播放、有"X5 控件"；样式不生效 | `<video x5-video-player-type="h5" x5-playsinline playsinline webkit-playsinline>` 几个属性都加；iOS 加 `playsinline` |
| 4 | **大图加载导致首屏白屏** | 婚纱照动辄 5MB+，首屏没图 | 必上 webp + lazyload；Hero 用 LQIP（低质量占位图）+ 渐进显示；CDN 必备 |
| 5 | **字体加载闪烁（FOIT/FOUT）** | 中文字体动辄 8MB+，加载完成前文字消失或瀑布回退 | 字体子集化（只取用到的字）；用 `font-display: swap`；衬线美感字体只用于 Hero，正文用系统字 |
| 6 | **微信内分享卡片缩略图** | 默认抓首图可能是 loading 占位 | 主动调 `wx.updateAppMessageShareData` 设置 imgUrl + title + desc |
| 7 | **iOS 长按图片菜单弹出** | 用户截屏/保存照片时弹出"添加到照片"菜单很烦 | `-webkit-touch-callout: none` 禁掉；或者主动提供"保存照片"按钮 |
| 8 | **横竖屏切换布局错乱** | 婚礼请柬一般竖屏设计，横屏体验崩 | 强制锁屏（`screen.orientation.lock` 或样式提示"请竖屏观看"） |
| 9 | **键盘弹出顶飞 fixed 元素** | RSVP 表单输入时，iOS 把 fixed 底栏顶上去 | 监听 `focusin/focusout` 切换 `position: fixed` ↔ `position: absolute`；或者用 visualViewport API |
| 10 | **微信 X5 不支持某些 CSS** | 部分 CSS 滤镜、backdrop-filter、CSS Houdini 不工作 | 上线前用 X5 调试器（chrome://inspect）真机调；高级特效准备 fallback |

## 六、关键发现 & 给设计师的建议

1. **页面控制在 6~8 屏**——超过 8 屏宾客会划疲劳；不重要的（住宿/Dress Code）放在底部 Tab，不要放进主流程。
2. **必须做 RSVP，但要做"轻 RSVP"**——国内场景不用 Joy 那么重的子事件回执；做"出席与否 + 人数 + 备注"三字段就够了，但留言/弹幕单独做。
3. **首屏一定有"开场动画"**——这是仪式感来源；推荐学 Greenvelope 信封而不是国内的"翻书"。
4. **照片是绝对主角**——Riley & Grey 和 Awwwards 案例都是"一张大照片 + 极简字"；不要做花哨叠层装饰。
5. **千人千面是低成本高回报的差异化**——只要从 URL 取一段 `?to=小明` 就能做到，让宾客感觉"专属"。
6. **避开五样"婚礼模板烂大街"元素**：3D 翻书、纸张飘落、烟花、烫金 LOGO 旋转、心形粒子。
7. **背景音乐必做但默认静音**——开场 + 引导按钮触发；提供一键关闭按钮（音乐图标常驻右上角，旋转时表示播放）。
