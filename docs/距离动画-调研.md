## 二、技术路线对比

| 方案 | 优 | 劣 | 适用 |
|---|---|---|---|
| **真实切片地图（高德/Mapbox）** | 准确、专业感、街道清晰 | 要 API key、需要服务、体积大、视觉风格难定制（要么科技、要么 GIS） | 强调"看到具体街道"的场景 |
| **矢量 SVG 地图（d3-geo + DataV GeoJSON）** | 无 API key、纯前端、矢量缩放无锯齿、颜色样式 100% 可控、数据精确 | 区/街道级别需多请求几个 GeoJSON | **平衡之选**：精确 + 视觉可控 |
| **抽象艺术化（纯 SVG 几何 + 装饰）** | 最轻、艺术感强 | 不真实、地标不准 → 与用户"想看到杭州→萧山"诉求冲突 | 浪漫优先、地理精确不重要 |
| **Canvas/Three.js 3D 地球** | 酷炫电影感 | 包大、移动端慢、不符合"暖色胶片"调性 | 高端品牌站 |

## 三、推荐方案（执行）

### 选定：**矢量 SVG 地图（d3-geo + DataV GeoJSON）+ GSAP timeline 镜头**

#### Why
1. **解决用户痛点**：用户明确想"看到杭州 → 萧山 → 两个小区"，矢量真实轮廓 = 真实，抽象方案做不到
2. **无 API key**：DataV 公开 CDN 直接拉，零依赖治理负担
3. **视觉可控**：所有 path 颜色/描边/滤镜都自己控，能完美贴合"米白 + 暖橘"调性
4. **轻**：只引 `d3-geo`（约 30KB gzip），GeoJSON 缓存 < 100KB
5. **手机端流畅**：纯 transform 镜头 + SVG 矢量 = GPU 友好

#### 架构
- 一个 SVG viewBox + 单一 `<g>` 镜头层，所有 transform 都在这层做
- 三组路径并存，按时间显隐：
  1. **杭州市全图**：`bound/330100_full.json`（含所有区轮廓，萧山在其中亮色高亮）
  2. **萧山区单独图**：`bound/330109.json`（单独详细轮廓）
  3. **两小区点 + 1km 连线**：根据真实经纬度（萧山中心约 120.27°E, 30.16°N），人工放两个点
- GSAP timeline：渐入杭州 → 高亮萧山 → 缩放进萧山 → pop 出两小区 → 1km 标尺浮现 → 文字层

#### 视觉细节
- 圆形舞台（保留），但用 `clip-path: circle()` 替代 border-radius，边缘更利
- 加 **羊皮纸纸纹背景**（CSS 渐变 + 极轻颗粒）替代纯色
- 地图轮廓用**双笔触**：外层粗淡橘 + 内层细深橘 → 老式胶片地图质感
- 杭州其它区域用**淡色填充**衬托，萧山用**暖橘**高亮
- 两个小区点：脉动光圈 + serif 中文标签（"他/她"）
- 1km 连线：strokeDasharray 渐次画出，中点浮现 "1.0 km"
- 终态文字：分层 stagger 入场，serif 中文 + 字间距宽松
